What it does

Send a PDF, photo or scan from any phone, laptop or tablet to the container via the standard LocalSend app — and find it indexed in Paperless-ngx a few seconds later. No taps, no confirmation, no manual upload.

1

Receive

The container announces itself on the LAN and auto-accepts every LocalSend transfer (optionally PIN-gated).

2

Forward

Each received file is POSTed to /api/documents/post_document/ with token auth and your configured metadata defaults.

3

Verify

The forwarder polls /api/tasks/ until Paperless reports SUCCESS, then deletes the local file.

4

Retry

If Paperless is down, the file is moved to /data/retry and re-attempted automatically.

How it works

Two processes run inside the container, supervised by a small tini-managed shell script:

  • localsend-cli recv — vendored Go binary from 0w0mewo/localsend-cli, built from source in a multi-stage Docker build and pinned to a release tag. It listens on TCP/UDP 53317, announces itself via UDP multicast (224.0.0.167), generates a self-signed HTTPS certificate at first run, and writes received files into /data/incoming. There is no UI to confirm transfers — every transfer is accepted automatically, optionally gated by a PIN.
  • python -m forwarder — small Python service that watches /data/incoming with watchdog (inotify), waits until each new file’s size stops changing, uploads it to Paperless-ngx using a token-authenticated multipart POST, and optionally polls /api/tasks/ to confirm consumption succeeded. On success the local file is deleted; on failure it is moved to /data/retry/<timestamp>/ with an .err sidecar and retried periodically.

Architecture

Requirements

  • A Linux Docker host. Host networking is required so LocalSend’s UDP multicast discovery (224.0.0.167:53317) reaches the container; Docker Desktop on macOS or Windows will not forward these packets properly.
  • A reachable Paperless-ngx instance and an API token (Profile → API Auth Token).
  • TCP and UDP port 53317 free on the host.

Quick start

git clone https://github.com/do2sz/localsend-paperless-docker.git
cd localsend-paperless-docker
cp .env.example .env
# Edit .env: at minimum set PAPERLESS_URL and PAPERLESS_TOKEN.
docker compose up -d --build
docker compose logs -f

Open the LocalSend app on a device in the same LAN. The configured alias (Paperless Inbox by default) shows up in the device list. Send any PDF, JPG or PNG — it should appear in Paperless within a few seconds.

Behaviour

VariableDefaultDescription

Volumes

  • /data/incoming — staging area for in-flight uploads. Normally empty between transfers.
  • /data/retry — files whose Paperless upload failed. Sidecar .err files hold the last error message. Files are retried automatically.

Both live under /data, which the bundled docker-compose.yml mounts to ./data on the host.

Operational notes

Discovery requires host networking. UDP multicast does not traverse Docker’s bridge networks, so without network_mode: host the container will be reachable only by typing its IP manually in the LocalSend app.
Self-signed certificate. The HTTPS certificate is generated and managed by localsend-cli on first start. LocalSend clients pin by SHA-256 fingerprint, not by trust chain — so this is fine.
Filesystem permissions. The container drops to UID 1000. If the bind-mounted ./data directory is owned by root on the host, the container cannot write to it. Fix once with sudo chown -R 1000:1000 ./data.

Troubleshooting

SymptomLikely cause / fix
Files pile up in /data/retry. Inspect docker compose logs and the .err sidecar. Most common: wrong PAPERLESS_URL, expired PAPERLESS_TOKEN, or PAPERLESS_VERIFY_SSL should be false.
Container is up but does not show in the LocalSend app. Host networking is not active, or the host firewall blocks UDP 53317 / TCP 53317.
Sender app rejects the connection. Wrong LOCALSEND_PIN, or the device is on a different VLAN where multicast is filtered.
Paperless rejects exotic formats. Install the optional Tika integration in your Paperless-ngx deployment to accept Office documents.
